Moving and Clustering Data with Sqoop and Spark SQL¶

What can machine learning tell us about flu rates around the world? Can we classify countries based on their relative flu rates? In this exercise we'll use an unsupervised learning technique called k-means clustering to cluster countries based on flu data. To do so, we'll use Apache Spark one of the most powerful tools available in the Big Data ecosystem.

As we explore the analytics techniques that Apache Spark offers us, we'd like to use some of our existing data. But that data is stored in a database. To get it out, we'll need to use a utility called Sqoop. Sqoop allows users to quickly move data to and from HDFS and relational databases.

Let's use sqoop to import our flu_statistics table into HDFS. This spawns a MapReduce job, so we need to tell MapReduce how to parallelize. For simplicity, we can use the region column of the table.

Spark SQL¶

Spark contains a library called Spark SQL which allows us to take advantage of Spark's scalable, parallel processing and SQL's intuitive, declarative approach to managing data. To use it, we'll create a Spark SQL context from our PySpark context.

In [2]: 
from pyspark.sql import *
sqlCtx = SQLContext(sc)

We now need to access our flu_statistics data in Spark. We've already used sqoop to move it to HDFS, so reading it into an RDD is as simple as calling the textFile method and splitting out the columns.

In order to apply Spark SQL to this data, we need to pair a schema (or description of the data) with the RDD. This means we'll need to do a few things:

  • Transform our lists of text into typed Row objects
  • Infer the schema based on the Row objects
  • Register a temporary table based on that schema

First, we transform to Rows. The transformation is simple: each entry in the RDD is fed into a Row constructor.

In [4]: 
flu_stats = flu_stats_text.map(lambda x: Row(region = x[0], country = x[1], year = int(x[2]), \
                                             week = int(x[3]), total_a = int(x[4]), total_a_and_b = int(x[5]), total_b = int(x[6])))

We can see that the flu_stats RDD is now full of Row objects.

We can now infer the schema of the RDD using the Spark SQL context. By registering the table, we can now refer to flu_stats in SQL queries.

In [6]: 
flu_schema = sqlCtx.inferSchema(flu_stats)
flu_schema.registerTempTable("flu_stats")

With Spark SQL getting the information we want to cluster on is as easy as it would be querying a database. We simply place our query inside the sqlCtx.sql command. For our clustering experiment we'll pull out the total type-A and type-B flu cases for each country.

In [7]: 
clustering_data = sqlCtx.sql("select country, sum(total_a) as a_sum, sum(total_b) as b_sum from flu_stats group by country")
clustering_data.take(10)

K-means and MLlib¶

The real power of Spark is not in simply being able to write SQL -- databases are better at that. What's important is that we can easily feed this data into a machine learning algorithm. As mentioned above, we'll be using a k-means clustering algorithm. In Spark, this simply means we need to import the correct class from pyspark.mllib.clustering, which holds a number of clustering algorithms.

In [8]: 
from pyspark.mllib.clustering import KMeans
from numpy import array
from math import sqrt
import pandas as pd
import numpy as np

We're going to create 2 RDDs for our machine learning case. The first will be a PairRDD which has a key and a value. The key is the country, and the value is an array consisting of the flu infection totals.

The second is a regular RDD which contains only the array of infection totals. We'll use it as the input to the k-means algorithm.

In [9]: 
clustering_input_pairs = clustering_data.map(lambda x: (x[0],array([float(x[1]), float(x[2])])))
clustering_input = clustering_data.map(lambda x: array([float(x[1]), float(x[2])]))
clustering_input_pairs.take(5)

Creating k-means clusters with Spark is very simple: we simply call KMeans.train with the input and the number of clusters. K-means will do its best to fit data to the number of clusters specified.

But how do we know how many clusters to make?

The short answer is we don't, so we have to guess. However, a good rule of thumb is to use the "elbow method." This means we evaluate range of values for the number of clusters and check the within set sum of squared error (WSSSE). Plotting the number of clusters against WSSSE will show improvement (minimization) somewhere along its progression. Choose the number of clusters at which large improvements in WSSSE stop.

To examine this, we'll simply evaluate the range of clusters from 2 to 10 and collect the results for WSSSE.

In [10]: 
clustering_error = []
for i in range(2,10):
  clusters = KMeans.train(clustering_input, i, maxIterations=10, \
        runs=10, initializationMode="random")
  def error(point):
    center = clusters.centers[clusters.predict(point)]
    return sqrt(sum([x**2 for x in (point - center)]))
  WSSSE = clustering_input.map(lambda point: error(point)).reduce(lambda x, y: x + y)
  print("Within Set Sum of Squared Error = " + str(WSSSE))
  clustering_error.append({"clusters":i, "wssse":WSSSE})

Now we're ready to make our final clusters of the flu data. As in our loop, we simply call KMeans.train.

In [14]: 
final_clusters = KMeans.train(clustering_input, 5, maxIterations=10, \
        runs=10, initializationMode="random")

Remember the PairRDD we created above? We can use that to quickly find out which countries belong to which clusters. The final_clusters object can be used in a map function to predict cluster membership based on an input vector. Here, the mapValues function allows us to produce a new PairRDD which has country as its key, and maps its values through final_clusters.predict to produce cluster membership.

In [15]: 
cluster_membership = clustering_input_pairs.mapValues(lambda x: final_clusters.predict(x))

We'd like to plot this data and understand cluster membership visually. As such, let's create a function which will map the result of a Spark join into an RDD of python dictionaries. We can then collect that into a DataFrame.

In [23]: 
def flatten_cluster_data(r):
    return {"country": r[0], "cluster" : r[1][0], "A": r[1][1][0], "B": r[1][1][1]}

complete_cluster_data = cluster_membership.join(clustering_input_pairs).map(flatten_cluster_data)
cluster_df = pd.DataFrame(complete_cluster_data.collect())

We can see that there are clear bands which separate some of the clusters. The UK, Japan and France are definitely different than the countries which are in other clusters. However, there are unseen factors that dominate our clustering results. Can you guess what they are?

The clusters on the right hand side of the graph are all developed and have large populations. It's likely that larger populations have higher total flu counts -- this naturally skews our clustering results. While it's also possible that some of the countries on the left hand side have more rigorous flu surveillance, we can't tell while population is clouding our view.

Try modifying the Spark SQL queries and re-run this work with per capita flu rates. How do cluster assignments change?

Summary¶

In this exercise we explored unsupervised learning using a number of Big Data tools. Specifically, we learned how to apply k-means clustering, and how to choose the appropriate number of clusters. Along the way, we discovered that population is probably obfuscating a simple clustering of countries by total flu cases -- a per capita approach is more suitable. We also touched on a number of pieces of in the Big Data Ecosystem:

  • We used Apache Sqoop to extract data from our database
  • We used Spark SQL to apply SQL queries to Spark RDDs
  • We connected Spark SQL to Spark MLlib to quickly add machine learning to SQL
